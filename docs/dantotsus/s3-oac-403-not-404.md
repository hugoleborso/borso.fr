# S3 + Origin Access Control hides "missing" inside "forbidden"

## Symptom

On the prod and preview CloudFront distributions, hitting any path
without a corresponding S3 object returned CloudFront's "Access
Denied" page (HTTP 403). The custom `errorResponses 404 -> /404.jpeg`
rule never fired — even on paths that obviously didn't exist (e.g.
typos in the URL bar).

## Root-cause chain

1. **Why?** CloudFront serves "Access Denied" instead of `/404.jpeg`.
   Because the origin (S3) returned 403, and CloudFront's
   `errorResponses` array only had a rule for 404, so 403 forwarded verbatim.
2. **Why does S3 return 403 for a missing key?**
   Because CloudFront's IAM principal had `s3:GetObject` only — no
   `s3:ListBucket`. Without `ListBucket`, S3 can't tell whether
   "object doesn't exist" or "object exists but you can't read it",
   so it picks the conservative answer (403) for both.
3. **Why does S3 conflate them by default?**
   Information-disclosure mitigation — telling unauthenticated callers
   "this key doesn't exist" leaks that other keys do exist.
4. **Why didn't `S3BucketOrigin.withOriginAccessControl(bucket)` grant
   `ListBucket` automatically?**
   Because it's an opinion call: most users don't want CloudFront
   probing for object existence. CDK grants the minimum required for
   content delivery (`s3:GetObject`).
5. **Why did we configure a `404` errorResponse without realising the
   chain wouldn't fire?**
   We pattern-matched on "S3 returns 404, CloudFront maps 404 →
   /404.jpeg" — the standard mental model for a public S3 bucket.
   With OAC, that mental model doesn't apply.

**Root cause:** we thought S3+OAC behaved like a public S3 bucket and
returned 404 for missing keys. With OAC's default permission set
(GetObject only) S3 returns 403 for both missing and forbidden keys,
and any custom `404 -> /404.jpeg` mapping is unreachable.

## Detection failure causes

- **Typing:** the construct's API doesn't model "missing-key
  behaviour"; nothing in the type signature would force the choice.
- **Linter / static analysis:** no rule for "if you set
  `errorResponses 404`, also set 403 OR grant ListBucket".
- **Functional validation locally:** we tested the happy path and the
  CloudFront distribution config; we didn't try a missing key.
- **CI:** unit tests assert structure of the synthesised template, not
  runtime behaviour against a real S3 origin. There is no integration
  test that hits a missing key.
- **Code review:** the OAC vs ListBucket interaction is non-obvious;
  reviewer would need to have hit it before to flag it.
- **Production monitoring:** no alarm on 403 rate or "Access Denied"
  page renders.

## Countermeasure

- **Code (per-app prod):** commit `16fde65` — `StaticSite.buildProd`
  attaches a second statement to the bucket policy granting
  `s3:ListBucket` to the CloudFront service principal, scoped via
  `aws:SourceArn` to the specific distribution:
  ```ts
  bucket.addToResourcePolicy(new PolicyStatement({
    actions: ['s3:ListBucket'],
    principals: [new ServicePrincipal('cloudfront.amazonaws.com')],
    resources: [bucket.bucketArn],
    conditions: { StringEquals: { 'aws:SourceArn': distribution.distributionArn } },
  }));
  ```
- **Code (shared previews):** commit `0f36781` — same pattern on the
  shared `previewsBucket` against `previewsDistribution.distributionArn`,
  plus an `errorResponses 404 -> /404.jpeg` entry on the previews CDN
  itself. The bucket-root `404.jpeg` is one shared fallback for every
  preview subdomain.
- **Operator action:** upload the fallback image once to
  `s3://borso-previews/404.jpeg`; CDK doesn't manage out-of-band
  content. Considered and rejected: a 403 → 404 remap, because it
  loses the semantic distinction between "forbidden" and "not found"
  and would still pop CloudFront's default page if the JPEG itself
  is missing.

## Eradication (rung 1 — structural impossibility)

- **Rung:** 1 (structural). Every CloudFront-fronted S3 bucket in
  this repo lives inside `StaticSite` (per-app prod) or
  `SharedStack` (shared previews), and both bake the
  `s3:ListBucket` grant into the bucket policy unconditionally.
  New apps using `StaticSite` get the grant for free; the
  misconception "I can serve a static site behind OAC and have
  missing keys return 404" is no longer expressible against our
  constructs.
- **What changed:** `infra/cdk/src/constructs/static-site.ts`
  appends a `PolicyStatement` granting `s3:ListBucket` to the
  CloudFront service principal, scoped via `aws:SourceArn` to the
  per-app distribution. `infra/shared/lib/shared-stack.ts` does
  the same for the previews bucket against the previews
  distribution. Both are accompanied by an `errorResponses
  404 -> /404.jpeg` rule that now actually fires.
