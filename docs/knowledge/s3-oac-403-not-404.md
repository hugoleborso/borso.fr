# S3 + Origin Access Control returns 403 (not 404) for missing keys

## Symptom

On the prod and preview CloudFront distributions, requesting any path
that didn't have a corresponding S3 object returned a CloudFront
"Access Denied" page (HTTP 403). The custom `errorResponses` rule
(`{ httpStatus: 404, responsePagePath: '/404.jpeg' }`) never fired.
Even paths with the trailing-slash convention (e.g.
`/some-missing-page/`) hit the same outcome.

## Root-cause chain

1. **Why** does CloudFront return 403 instead of 404?
   The origin (S3) returned 403, and CloudFront's `errorResponses`
   array has no rule for 403 → so it forwards 403 verbatim.
2. **Why** does S3 return 403 for a missing key?
   With Origin Access Control, CloudFront's IAM principal only has
   `s3:GetObject` on `arn:aws:s3:::bucket/*`. To distinguish
   "object doesn't exist" from "object exists but you can't read
   it", S3 needs `s3:ListBucket` on the bucket ARN itself. Without
   that, S3 chooses the conservative answer (403) for both cases —
   it's an information-disclosure mitigation.
3. **Why** doesn't `S3BucketOrigin.withOriginAccessControl(bucket)`
   grant `ListBucket` automatically?
   It grants the minimum required for content delivery
   (`s3:GetObject`). `ListBucket` is an opinion call: most users
   don't want CloudFront probing for object existence, so CDK leaves
   it off by default.
4. **Why** wasn't this caught earlier?
   `errorResponses 404 -> /404.jpeg` looks correct in code review and
   passes synth-level tests. The only failure mode is a missing-key
   request, which only happens once the site is live and visitors
   start clicking broken links.

**Root cause:** S3's privacy-preserving 403-for-everything default,
combined with CloudFront forwarding the origin's status untouched
when no `errorResponses` entry matches.

## Fix

- **Code (per-app prod):** commit `16fde65` — `StaticSite.buildProd`
  attaches a second statement to the bucket policy:
  ```ts
  bucket.addToResourcePolicy(new PolicyStatement({
    actions: ['s3:ListBucket'],
    principals: [new ServicePrincipal('cloudfront.amazonaws.com')],
    resources: [bucket.bucketArn],
    conditions: { StringEquals: { 'aws:SourceArn': distribution.distributionArn } },
  }));
  ```
  The `aws:SourceArn` condition keeps the grant scoped to this
  specific distribution.
- **Code (shared previews):** commit `0f36781` — same statement on
  the `previewsBucket`, scoped to `previewsDistribution.distributionArn`,
  plus an `errorResponses 404 -> /404.jpeg` entry on the previews
  CDN itself. The bucket-root `404.jpeg` is one shared fallback for
  every preview subdomain.
- **Operator action (previews bucket only):** upload the fallback
  image once; CDK doesn't manage out-of-band content:
  ```bash
  aws --profile borso-admin s3 cp 404.jpeg s3://borso-previews/404.jpeg
  ```
- **Why not 403→404 remap instead?** Considered and rejected — it
  loses the semantic distinction between "forbidden" and "not found",
  and it would still pop the CloudFront default page if the JPEG
  itself is missing. Granting `ListBucket` keeps the chain honest.
