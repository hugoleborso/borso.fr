## Notes

- **#03**: Mobile selector progresses one column at a time and exposes a Back button, but no numeric step indicator (e.g. "Step 2 of 3") is rendered. The column heading change + Back covers user navigation; consider whether the spec intent ("step indicator") was satisfied by the heading swap or if explicit progress numbers are desired.
- **#04**: Start button label is `Start session` everywhere. Spec step 2 explicitly states the label changes to `Drill this variation` once a variation is selected. This is a verbatim mismatch with the spec.
- **#11**: The cleared banner is correctly an inline non-modal panel and contains exact spec text. **But it is positioned BELOW the board** (DOM sibling order `board-container` then `inline-banner` inside `.board-area`), forcing the user to scroll on a 1280×800 viewport. Spec sentence 7 says "an inline banner appears above the board". Decide whether to reorder or amend the spec.
- **#12**: After clicking `Switch to Play with this scope`, the resulting Play scope pills displayed `Modern Defense / Main Line / Modern Defense`, not `Italian Game / Main Line / Italian Game C50` (the variation just drilled). The transition fired and Play mode entered, but the scope mapping appears to fall back to the first opening with a matching variation name. Worth a follow-up; not a hard fail of the asserted line.
- **#13**: `LoadingPanel` source exists and is wired in `App.tsx`; runtime visual capture is infeasible because the localhost dev server returns the bundled `openings.json` in <130ms, faster than any single screenshot can be taken. Logic verified by source.
- **#14**: White's first move was 1.d4 in this run (drilled scope was Modern Defense after the Switch-to-Play scope migration). The point — White auto-plays before the user can interact — is verified regardless of which book move is selected.
- **#18 / #19 / #20**: Bug-list rows B2, B5, B6 require interaction patterns (drag-and-drop on the board, multi-step scope-switch races, Reset+arrow-recompute timing) that would consume more screenshot/eval budget than this run allows. Recommended for the next validation run with a relaxed cap.

## Verdict: FAIL

- 2 PASS_WITH_NOTE rows (#11 banner position, #12 scope migration after Switch-to-Play) that warrant operator attention but did not violate the asserted line outright.
- 3 UNVERIFIABLE (B2, B5, B6) due to screenshot/test budget — flag for follow-up validation.
- All other 17 assertions PASS, including the merge-gate input metric (Pick variation → Drill cleared → Switch to Play → no OOB observed) for the Italian Game / Main Line scope.
