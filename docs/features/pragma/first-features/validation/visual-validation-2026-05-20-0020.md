## Notes

### FAIL rows

- **Row 10 (catalog list — energy badge + mastery aggregate).** Spec dispatch brief is explicit: "list shows title, composer, energy badge, mastery aggregate". The shipped card shows title, artist, status, tonality. The `baseEnergy` field is stored on the song (verified — set to 6 for Take Five) but not surfaced on the list card. The mastery aggregate is computable from `mastery.core.ts` (impl-03 verdict says it ships at 100% coverage) but no UI calls it. Two missing surfaces, one row.
- **Row 11 (chord chart viewer + Mode Scène).** Spec *Designer-pass resolutions* §5: "inline preview on the song detail page; tap to enter fullscreen Mode Scène (black background, large chord grid, auto-scroll, A−/A+ zoom, keyboard nav, song-pill carousel of the setlist)". None of this is implemented. The ChordPro text is shown as a plain `<textarea>` on the edit form; there is no preview/render mode, no fullscreen toggle, no Mode Scène route. This is the single biggest spec-shipping gap.
- **Row 15 (concert detail — gear + friendsCountPerMember).** Spec section 3 use case 1: "User creates a `Concert` with date / venue / capacity / gear" and step 2: "Each band member opens the concert, fills in their `friendsCountPerMember` field". The DB schema and API contract both support these (verified via `GET /api/sessions/<id>`), but the front-end concert detail page renders only `Venue` + `Capacity` as a read-only `<dl>` — no `gear` field, no per-member friends-count grid, and no edit form at all. The concert was created via "+ Concert" with defaults (`TBD` venue, 0 capacity, today's date) and there's no way to update those values from the UI. Practical impact: the concert detail page is barely usable as a concert plan.
- **Row 16 (practice ↔ concert linkage).** Spec section 3 use case 3: "User creates a `Practice` and links it to the upcoming concert via `preparedConcertId`". The DB column exists and is nullable; the API accepts it; but the UI exposes no selector, no field, no visible link from practice → concert. The practice detail page shows only `Practice — <date>` + the Setlist editor.
- **Row 19 (stale-bar banner).** Spec section 4 use case 3: "At login, the app surfaces an in-app banner for bars with no interaction for >N days (N to define with designer; default 60)". Cafe Pop was set with `lastInteractionAt = 2025-12-01` (≈170 days stale at the test date 2026-05-20). On `/bars` reload, no banner is rendered anywhere in the DOM. Either the threshold logic is not implemented or the surface is missing.

### UNVERIFIABLE rows

- **Row 12 (mastery matrix — 5×7 + row/col averages).** Only 2 members × 3 instruments seeded in our test data; cannot exercise the spec's specific "5 × 7 grid" structural claim without the band's actual roster. Row and column averages mentioned in spec section 1bis ("Row averages and column averages update live") are not visibly present in the matrix UI we observed; flagged for closer look but not failing because the assertion is about behaviour at a data scale we did not reach.
- **Row 13 (mobile drag-via-handle pattern).** Spec Q.O.D. pins `mobile drag pattern: handle`; shipped UI uses `↑` / `↓` arrow buttons on each row. Impl-03 kaizen acknowledges the deferral ("drag-via-handle is genuinely v2 polish; buttons cover the same intent today"). Functionally equivalent for desktop + a11y-good alternative; spec compliance on the *pattern* is grey-zone, not strictly PASS. The kanban DnD on bars is full HTML5 DnD (PASS).
- **Row 23 (offline writes fail with clear UI).** Verified the "OFFLINE — READ ONLY" banner; did not exercise a write attempt while offline. The spec's UX intent is the banner, which is present, but the failure-mode is not exercised end-to-end.
- **Row 24 (FR default).** Same round-1 ambiguity. Without a runtime switcher or a `--lang` flag that survives navigator-detection, the FR catalog cannot be exercised in this validator's browser. The catalog file itself is correct (read directly: `auth.title = "Pragma — espace du groupe"`); the unit-tested `detectInitialLocale('fr-FR') = 'fr'` confirms an FR-locale browser would render FR. **Spec sentence remains ambiguous between "default FR catalog" (current impl) vs "default FR render" (literal reading).** Worth one sentence in the Q.O.D. row.
- **Row 26 (i18n switcher).** No reachable switcher. Same shape as round 1.

### Visible deviations from the design bundle

- **Member chips colour.** All chips render with the accent blue (`#2d5fa0`) because no per-member `color_hex` was set during create. The design bundle shows distinct coral/teal/mustard/plum/sage hues. The CSS tokens are defined (verified — `--member-coral` etc. all present); the wiring of "new member → assign next palette slot" is the missing link. Not failing because the spec also says `color` is a field the admin sets (which we did not set), but worth a UX polish: pick a palette entry by default rather than fall back to accent.
- **Energy curve sparkline.** Renders `M …` (move-only) path because both entries have null energy. Filling the energy slider on each entry would draw the line. Spec is correct (spec section 2 step 6 says "User fills in energy 1-10 per entry; the curve renders below"); we did not exercise the slider, so the sparkline is in its empty state. Not failing.
- **Status enum casing.** Catalog list shows status as upper-case ("IDEA", "CONTACTED", "BOOKED"); database holds them lowercase. Cosmetic — `text-transform: uppercase`. The casing is consistent across surfaces (bars list/kanban also uses upper-case).

5 FAIL rows (10 catalog card contract, 11 chord chart + Mode Scène, 15 concert gear/friends, 16 practice linkage, 19 stale-bar banner), 5 UNVERIFIABLE rows (12 mastery 5×7, 13 mobile handle, 23 offline writes, 24 FR default, 26 i18n switcher), 18 PASS rows.

The shipped surface implements the core CRUD + offline + dark-mode + transition warning + sparkline + kanban DnD cleanly. The five gaps are:

1. **Chord chart viewer + Mode Scène** — the single most visible feature in the design bundle (§5 of Designer-pass) is missing. The ChordPro text is captured but never rendered.
2. **Concert edit surface** — gear, friends-count-per-member, and an edit-form for the date/venue/capacity are all absent. The concert detail page reads back what was created at "+ Concert" time, but cannot be amended.
3. **Practice → Concert linkage** — `preparedConcertId` is in the data layer but has no UI.
4. **Stale-bar banner** — spec section 4 use case 3 surface not shipped.
5. **Catalog card extras** — energy badge + mastery aggregate columns absent from list cards (small relative to the others).

Per the visual-validation standard: **FAIL is never mergeable**. The orchestrator should advance to an implementation slice that covers the five FAIL rows before re-running this validation. The UNVERIFIABLE rows (FR default + handle drag + i18n switcher + offline write attempt + mastery 5×7) are spec/data-scope gaps and do not block on their own — they should be disclosed in the PR description if and when the FAIL rows are fixed.

## Kaizen — visible surface (round 2)

- **Chord chart preview is the highest-ROI follow-up.** The design bundle's §5 Mode Scène is the single most distinctive feature in the spec and the one a reviewer would test first. Inline preview + fullscreen toggle ≈ a one-week PR if styled to match the design bundle.
- **Concert detail needs an edit form.** Today there is no way to set venue / capacity / gear / date / friends-count after the "+ Concert" insert. The data model is ready; only the UI is missing.
- **Stale-bar threshold lives where?** A single `?stale_after_days=60` query param on `GET /api/bars` (or a client-side filter on `lastInteractionAt`) would make the banner trivial — gate is a 30-line component + a one-line constant.
- **Member chip palette wiring.** The CSS tokens (`--member-coral`/teal/mustard/plum/sage) are defined but unused on freshly-created members (all fall back to accent blue). A round-robin assignment at member-create time, or a colour picker in the create form, would unlock the design bundle's intent.
- **Mode Scène alternative if the full feature slips:** even a CSS-only chord-text preview (rendered ChordPro highlighting on `<pre>` blocks) would close the "the ChordPro text is captured but never rendered" gap without committing to the fullscreen viewer.
- **Same as round 1, still open:** manifest icons remain `[]`; no globally-exposed i18next handle for dev/test; no `?lang=` URL escape hatch for the validator.
- **Wholly new (round 2):** the dev server's `concurrently` orchestration exits because `dev:db` returns immediately after printing the DB URL. `concurrently -k` then kills the other two. This is a one-line fix (have `dev:db` `exec` into a wait loop, or drop it from `concurrently` and document the manual `pnpm run db:local:start` step). The validator burned ~5 minutes booting the API + Vite manually because of this.
- **Wholly new (round 2):** the bars-form Save click registered DOM-wise but the form submission didn't trigger the API POST until `form.requestSubmit()` was called via `eval`. May be an event-handler attached to the wrong element (button vs form). Reproduces every time on the bars form; the songs form is fine.
