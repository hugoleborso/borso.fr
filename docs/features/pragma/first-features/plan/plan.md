## Open questions / unknowns

Items the spec is genuinely silent on and that need a tech-lead answer before `/implementation`. Strikethrough items have been pruned by the tech-lead as resolvable from the spec + design bundle + neighbouring workspace (`last-loop-lepin`) without a separate decision — they are not real ambiguities, they are documented choices that belong inside the plan rows themselves or follow obvious repo precedent.

- **Initial `mastery_default` rows on first deploy.** The spec says "manual UI" for the DB seed. It does not say whether the 5 × 7 = 35-cell matrix on `/members` renders empty (no rows in the table yet) or pre-populated with 35 explicit-zero rows. The choice is UX-visible: clickable-empty vs explicit-zero. Implementation default if unanswered: render the 35 cells from the cartesian product, write a row only when the user edits a cell. Confirm at `/implementation`.
- **PWA manifest icon set.** Design bundle ships no icons. The PWA manifest needs at minimum 192 × 192 + 512 × 512 PNG to install on iOS / Android. Implementation default if unanswered: generate placeholder icons from the blue accent + `P` wordmark, flag for replacement before prod.
- **Members admin lifecycle.** Spec lists `/members` admin but does not state add / remove / soft-delete semantics, and the consequence for `mastery_default` + `mastery_override` rows referencing a departing member is undecided. Default if unanswered: hard-delete with `ON DELETE CASCADE` on the two mastery tables and on `default_lineup` JSONB key cleanup; flag in the `/implementation` verdict if this turns out to lose history Hugo wanted to keep.
- **`isHarmonic` flag default for new instruments.** Spec is silent on what value the admin form pre-selects. Default if unanswered: `false`; admin opts in on piano / guitar / accordion / synth. Low blast radius — only affects the transition-warning algorithm for newly-added instrument types.

## Missing technical skills

The bar is *"the planner faced a non-obvious choice in this domain that a sub-skill recipe would have answered cleanly"*, not *"we touch this tech area"*. Two entries cleared the bar; four did not (`last-loop-lepin` is the working precedent for Vite, Drizzle, Hono, and CDK shape — no new recipe needed for those).

- **`/pwa-service-worker`** — first PWA in the repo. The planner had to invent the cache strategy from scratch: which assets to precache vs runtime-cache, how to version the cache, how to expire the next-session setlist when the upcoming session date changes, how to surface a "service worker updated, reload to apply" prompt without breaking the offline experience. A recipe would have shortcut the design.
- **`/react-i18next`** — first i18n surface in the repo. The planner had to invent the catalog file layout (namespaces vs flat), the key-parity test, the dev-time missing-key fallback, and the default-locale-detection logic. Future apps adopting i18n will re-invent unless captured.
