## Notes

- **Row 11 (FAIL).** `RunnerAdminPanel.tsx` contains no `<input type="file">` or any photo-related state; the back end exposes `POST /api/admin/media/presign` (per `app.ts`) but the front end never wires it. Spec requires « nom, photo (upload pré-course OU selfie live le jour J ; fallback initiales) ». Only the fallback path (initials avatars) is wired. The Gemba scenario "orga selfie le jour J" is unreachable from the UI.
- **Row 15 (FAIL).** Spec « L'orga peut télécharger un CSV/PNG récap » is not implemented anywhere reachable from the UI. No download button on the spectator finished-race view, none on the admin panel. Backend has no `*.csv` / `*.png` endpoints either (`app.ts` route list is exhaustive).
- **Row 17 (FAIL).** Spec « Erreur de pointage : l'orga peut éditer/supprimer un pointage. Une bannière publique […] apparaît sur la vue spectateur pendant 60 s. » — the banner triggers on `PUT` (correction) but not on `DELETE` (annulation). After annulating Alice's B3 via the Corrections tab, the API returned the voided punch but `/api/standings` `mostRecentCorrectionAt` stayed `null` and no banner appeared on `/`. Either `voidedAt` should also feed into `mostRecentCorrectionAt`, or the spec's banner contract for deletions should be revised. Currently a supporter watching from afar sees an eliminated runner suddenly drop off the leaderboard with no public signal that a correction happened.
- **Row 10 (PASS_WITH_NOTE).** The Setup tab is read-only once an edition is `live` or `finished`; the upload/creation form is gated by `currentEdition.status === 'setup'` in `SetupPanel.tsx`. For an organiser who needs to fix a typo on the GPX or shift the end time post-creation, there is no recovery path in the UI other than wiping the database. Defensible v1 behaviour (one edition / year, one shot) but worth a follow-up.
- **Row 21 (PASS_WITH_NOTE).** API-level conflict (409 on duplicate `editionSlug + runnerSlug + loopIndex`) confirmed by curl. The "déjà pointé à HH:MM:SS" UI message was not exercised — it would need two concurrent admin sessions clicking the same row. Spec-level acceptable; flagged so the next round can race two browser tabs.
- **Row 22 (UNVERIFIABLE).** The brief itself stated « The archives entries are still text-only (no click target on each line). Treat as a minor cosmetic gap, not a structural FAIL. » Confirmed by reading the code (`SpectatorPage.tsx:76`) — the archives list is rendered when the current edition is `setup` and there exists at least one `finished` past edition. With only one finished edition seeded and no upcoming edition, the archives card is unreachable from the running dev stack, so the click-target gap cannot be observed in the browser this run.
- **Row 23 (PASS_WITH_NOTE).** GPX entry uses a `<textarea>` paste flow rather than an `<input type="file">` upload. Both achieve the same functional outcome (back end parses the raw XML), and the spec word "upload" can reasonably cover paste. Flagging so it stays visible.
- **Row 20 (UNVERIFIABLE).** Triggering 5 wrong PINs in 5 min would lock out the admin IP for the rest of the run and block every subsequent admin tab walk. The login + rate-limit contract is asserted by Gate B (`auth.middleware.test.ts`); deferring to that gate is the safer trade.

## Verdict: FAIL
