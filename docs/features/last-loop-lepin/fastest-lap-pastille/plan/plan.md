## Open questions / unknowns

**§A. Should the client `.optional().default([])` be tightened to `z.array(...)` (non-optional) once both halves have shipped?**
The `.optional().default([])` exists for the deploy gap (server omits the key briefly during rollout). Once `fastestLap` is part of every served response, the `.optional()` becomes dead. **Resolution adopted in this plan: keep `.optional().default([])` indefinitely.** Tightening it is a Zod-strictness tradeoff with no clear win — the `default([])` makes the client robust to a future bug where the server-side projection regresses to "field missing". **No blocker.**

**§B. Where does each `<Leaderboard>` callsite get its `StandingsDto` from?**
Four callsites identified (`SpectatorPage.tsx`, `AdminPage.tsx`, `RunnerFichePage.tsx`, `ArchivesPage.tsx`); the first three use `useStandings(editionSlug)` returning a `StandingsState` (`standings: StandingsDto | null`); the fourth uses `useResource(...)` with `apiClient.getStandings`. In every case the parent already has access to `standings.fastestLap` directly — no extra plumbing. **No blocker.**

**§C. Should `fastest-lap.core.ts` accept `LoopPunch[]` already filtered for `voidedAt === null`, or filter internally?**
`ranking.core.ts` already computes `validPunches` and passes it forward to other helpers. Symmetric choice: `fastest-lap.core.ts` accepts the **unfiltered** list and filters internally — this is the safer default (a caller that forgets to filter doesn't silently include voided punches in the record). The test vector "all-voided returns `[]`" covers the filtering. **No blocker.**

**§D. Inline SVG vs CSS-only badge?**
Spec says "icône chronomètre SVG en blanc". Two implementations: inline `<svg>` inside the badge `<span>`, or pure CSS (background-image data-uri, or a `::before` with a glyph). Resolution adopted: **inline `<svg>`** — keeps the badge accessible (an inline SVG can carry `<title>Fastest lap</title>` for screen readers, which a CSS background-image cannot), and matches the repo's existing pattern (no SVG asset files under `site/src/`, no SVG data-URIs in CSS). **No blocker.**

**§E. Concurrent sister-agent on `self-punch-runners` is editing `vitest.workspace.ts`.**
TL brief says the sister agent extends the workspace globs. This plan does **not** touch `vitest.workspace.ts` — `fastest-lap.core.ts` is gated by the existing `api/src/**/*.core.ts` rule. **No blocker; pure file-surface disjointness.**
