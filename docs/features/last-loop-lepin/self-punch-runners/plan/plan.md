## Risk register

| Risk | Severity | Mitigation in plan | Detection if it slips |
|---|---|---|---|
| **R-DSQL** : `ALTER TABLE loop_punches ADD COLUMN source TEXT NOT NULL DEFAULT 'admin'` peut être rejeté par DSQL (la doc `dsql-postgres-compat-gaps.md` ne le couvre pas) ou peut accepter la statement mais ne pas matérialiser le default sur les rows existantes, laissant `source` NULL et violant la contrainte au prochain read. | High | Dry-run local (Postgres réel via `local-postgres.sh`) **et** dry-run sur l'environnement preview avant prod. Si rejet ou rows NULL : remplacer par 3 statements séparés (`ADD COLUMN nullable` → `UPDATE … SET source='admin'` → `ALTER COLUMN SET NOT NULL`), que le migration runner split déjà via `statement-breakpoint`. | Le `_migrations` row n'est pas inséré (la migration échoue à mi-chemin) ; l'app boot Sentry-erreur sur le premier query. Alerte > 1 occurrence. |
| **R-FSM** : un état de la state-machine `SelfPunchModal` est oublié, un tap finit sur un écran vide. | Medium | FSM 100 % couverte côté `*.utils.ts`. Test unitaire énumère explicitement les 9 transitions ; le `default` du reducer throw `assertNever`. | Test unitaire `noUncheckedIndexedAccess` ; runtime, le throw remonte en Sentry. |
| **R-IP-leak** : oubli de retirer la lecture d'IP côté controller. | Low | Aucune colonne `ip` en schéma — le code n'a pas où écrire l'IP même si quelqu'un essayait. | Schéma drizzle ne contient pas `ip` ; biome lint + revue. |
| **R-Acc-100m-edge** : un coureur à exactement 100 m est rejeté (strict `<`). Comportement intentionnel mais asymétrique entre client et serveur si l'un utilise `<=` par mégarde. | Low | Constante partagée + même comparateur des deux côtés. Test : position à exactement 100 m ⇒ rejet côté serveur ET client (assertion alignée). | Test unitaire haversine sur point synthétique à 100 m ; revue manuelle du diff sur les deux call sites. |
| **R-Geo-timeout** : `getCurrentPosition` peut tourner 30+ secondes en cellulaire. Le coureur reste sur la modale "Je localise…" sans feedback. | Medium | Timeout 10 s côté `requestPosition.utils.ts`, géré par la FSM (`timeout` ⇒ modale erreur avec retry). | Test unitaire avec `vi.useFakeTimers()` + advance ; visual-validation simule un position request qui ne resolve jamais. |
| **R-DOM-button** : transformer la chip en `<button>` peut casser un style ou capturer un focus indésirable sur l'écran de retransmission (qui n'est pas tactile mais peut recevoir le focus si une télécommande s'égare). | Low | Conserver les classes CSS existantes. `type="button"` (pas submit). Pas de `autoFocus`. | `/visual-validation` regarde le rendu pixel-similaire au pré-feature ; lint Biome a11y. |
| **R-Drift-haversine** : les deux `haversine.utils.ts` divergent silencieusement après une modification d'un seul côté. | Low | La formule haversine est mathématiquement gelée (seul `EARTH_RADIUS_METERS` est en jeu, et il ne bouge pas). Les vecteurs de test partagés (Paris-Lyon, antipodaux, intra-100m) sont identiques des deux côtés ; un drift dans la formule fait diverger les distances calculées et fait casser le test back-e2e qui POST une position et lit la `distance_from_center_m` retournée par le serveur (cf. *self-check Q7*). | Test back-e2e qui compare la distance calculée client (assertée via le retour serveur) à la distance attendue ; si la duplication drifte, les deux ne matchent plus. |
| **R-Geofence-startLatLng-mauvais** : la `gpx.startLatLng` du GPX peut ne pas être le point de départ-arrivée physique (premier point du tracé GPX, qui peut être à 50 m du panneau "DÉPART"). | Medium | 100 m de rayon absorbe un offset de quelques dizaines de mètres. Smoke test sur place avant le jour J : ouvrir `/`, tap sa propre chip au point d'arrivée physique, vérifier que la position est dedans. | Pré-prod : un opérateur fait le test physique. Production : Sentry `self_punch_rejected_client_out_of_zone` > 10 / 10 min ⇒ alerte (déjà dans le spec). |
| **R-Couverture-cellulaire** : pas de réseau au point d'arrivée, le POST échoue. | Medium (vu inconnu) | Modale erreur réseau + bouton retry, et l'admin reste filet de sécurité (déjà acté dans le spec). | Pré-prod : test cellulaire sur place avant le jour J. Si KO : annoncer aux coureurs que l'admin pointera ce jour-là. |
| **R-RTL-tests-front** : le spec a mentionné `*.test.tsx` testing-library, mais le repo ne ship pas testing-library et la convention site est *uniquement* `*.utils.test.ts`. | Low | Décision plan-time : pas de tests RTL. La FSM va dans `selfPunchModal.utils.ts` (couverte), la React shell est validée par `/visual-validation`. | Coverage gate `site/**/*.utils.ts` à 100 % couvre la logique ; `/visual-validation` couvre le rendu. |

## Pattern Coherence pass

- **Deps inventoriées** : `react@19`, `hono@4`, `drizzle-orm@0.36`, `postgres@3.4`, `zod@3.24`, `@hono/zod-validator@0.4`, `leaflet@1.9`. Aucune nouvelle dep introduite par cette feature (l'haversine est pure-TS, le `getCurrentPosition` est un global navigateur). **OK.**
- **Patterns d'état** : `useState` pour les composants admin (`CorrectionPanel`, `SetupPanel`), `useEffect` *uniquement* dans `CourseMap` (justifié — bridge Leaflet) et `clock-store.ts` (justifié — `useSyncExternalStore`). Cette feature reste sur `useState` + event handlers ; aucun nouvel `useEffect`. **OK.**
- **Pattern modal** : aucun composant `Modal` existant dans le repo. Cette feature en introduit un. Pas de "absorption" possible ; la cohérence se mesurera la prochaine fois qu'on ajoute une modale.
- **Pattern controller public sans auth** : `*.controller.ts` actuels sont tous admin (`requireAdminSession`). Le seul "public" actuel est l'edition controller (read-only). Cette feature ajoute le premier *write* public. Convention nouvelle ; nommage `self-punch.controller.ts` la marque visuellement.

## Open questions / unknowns

1. **DSQL `ADD COLUMN ... NOT NULL DEFAULT`** : non documenté dans `dsql-postgres-compat-gaps.md`. Le plan prévoit un fallback en 3 statements si le dry-run échoue, mais le verdict ne sera connu qu'au gate 7. À-confirmer avant de lancer la migration en prod : si le dry-run preview révèle un échec, on bascule au fallback avant de lancer le déploiement prod. *Non-bloquant pour `/implementation` — le fallback est dans le plan.*
2. **Coexistence avec la card admin "DNF à valider" du spec parent** : self-punch n'écrit jamais un DNF, donc la card admin reste inchangée. Aucun changement dans le composant `admin/*`. Confirmation visuelle attendue dans `/visual-validation`. *Non-bloquant.*

> Aucun blocker pour `/implementation` après la résolution de la question haversine (décision plan-time révisée : duplication front/back, cf. Q8bis et R-Drift-haversine).
